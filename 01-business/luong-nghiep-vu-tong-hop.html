F01

Khách đăng ký / đăng nhập tài khoảnkhách

Domain: authentication (customer-auth-service) · Bề mặt: customer app (Flutter) + web (Next.js)
  1. Khách làm gì: Mở app/web → đăng ký bằng SĐT hoặc email → nhận OTP (6 số) → nhập OTP → tạo mật khẩu. Lần sau đăng nhập bằng SĐT + mật khẩu, hoặc Đăng nhập với Google.
    ⚠️ Kênh gửi OTP (đã xác minh): email → gửi qua Email (Brevo); số điện thoại → gửi qua Zalo ZNS. KHÔNG có SMS.
  2. Đường đi: App/web → BFF (/api/auth/login, register, verify-otp, google-login) → gateway → customer-auth-service (8081). Gateway giới hạn đăng ký (2 lần/phút). Bảng: customers. ResponseCode "200".
  3. Socket: Không.
  4. Admin: Khách mới xuất hiện trong danh sách khách (F12) sau khi đăng ký. Admin không dùng luồng này — admin login riêng qua internal-auth (F15).
sequenceDiagram
    participant K as Khach (app/web)
    participant G as Gateway
    participant A as customer-auth
    K->>G: Dang ky (SDT/email)
    G->>A: register
    A-->>K: Gui OTP (Email hoac Zalo ZNS - KHONG SMS)
    K->>G: Nhap OTP + tao mat khau
    G->>A: verify-otp
    A-->>K: Thanh cong, cap token
    K->>G: Dang nhap (SDT+MK / Google)
    G->>A: login
    A-->>K: access + refresh token

Khách bổ sung KYC (CCCD/bằng lái) → Admin duyệtkhách+admin

Domain: customer (KYC) + storage (xanh-service) · Bề mặt: customer app + web
  1. Khách làm gì: Màn Định danh → chụp/tải CCCD trước+sau, nhập số/ngày hết hạn. Màn Bằng lái → tải ảnh, nhập hạng/số/ngày. Trạng thái: Chờ duyệt / Đã duyệt / Từ chối (kèm lý do).

Đổi xe do sự cố (lỗi kỹ thuật / tai nạn)khách+admin (mới)

Domain: booking (incident + change-car) (xanh-service) · Bề mặt: web admin + Flutter admin
Phân biệt với F08 (đổi xe tự nguyện trên ONGOING). Luồng này dành cho xe gặp sự cố giữa chừng: đi qua trạng thái INCIDENT (7) rồi mới đổi xe. Đã có và khá đầy đủ trong code (state machine 9 trạng thái).
  1. Khách làm gì: Xe đang thuê gặp lỗi kỹ thuật hoặc tai nạn → khách báo hotline / nhân viên hiện trường (không tự thao tác hệ thống).
  2. Đường đi (3 bước):
    • B1 — Báo sự cố: update-status ONGOING (3) → INCIDENT (7), bắt buộc kèm incidentType = TECHNICAL_FAULT (lỗi kỹ thuật) hoặc ACCIDENT (tai nạn). Sai/thiếu type → reject.
    • B2 — Đổi xe (văn phòng): change-car INCIDENT (7) → CHANGING_CAR (9), chốt xe mới + tiền chênh lệch. Log CHANGE_CAR_OFFICE. (Chỉ đổi xe được khi đang INCIDENT.)
    • B3 — Giao xe mới (hiện trường): change-car/deliver → giao xe thay thế, quay lại ONGOING (3). Log CHANGE_CAR_DELIVERY.
    Bảng: new_bookings (status + incidentType), booking_logs, booking_car_history.
  3. Socket/Thông báo: khi báo sự cố, code publish NotifyEvent.IncidentRaised (Telegram) — nhưng hiện chỉ BookingCreated đang bật nên sự cố chưa gửi thông báo thật.
  4. Admin xử lý: Báo cáo có bộ lọc trạng thái sự cố (7/8). Ngoài đổi xe, từ INCIDENT còn có thể: báo nhầm → ONGOING (3); kết thúc HĐ sớm → REFUND_PENDING (6) (perform-receive chấp nhận cả INCIDENT); tịch thu xe → SEIZED (8).
